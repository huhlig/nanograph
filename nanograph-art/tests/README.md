# Adaptive Radix Tree Integration Tests

This directory contains comprehensive integration tests for the Adaptive Radix Tree (ART) implementation.

## Test Files

### art_tests.rs

Basic unit tests covering core ART functionality:
- Empty tree operations
- Single and multiple insertions
- Updates and deletions
- Key existence checks
- Common prefix handling
- Node growth and shrinking
- Iterators (full, keys, values)
- Binary keys and values
- Large datasets
- Path compression
- Memory usage tracking
- Clone and default implementations

### integration_tests.rs

Comprehensive integration tests covering:

#### Basic Operations Tests
- **test_kvstore_basic_put_get**: Basic put/get operations
- **test_kvstore_update_existing_key**: Key updates
- **test_kvstore_delete**: Deletion operations
- **test_kvstore_multiple_operations**: Batch operations

#### Range Query Tests
- **test_kvstore_range_query_full**: Full range scans
- **test_kvstore_range_query_bounded**: Bounded range queries
- **test_kvstore_prefix_scan**: Prefix-based scanning

#### Concurrent Operations Tests
- **test_concurrent_inserts**: Parallel insertions
- **test_concurrent_reads_writes**: Mixed parallel operations

#### Stress Tests
- **test_large_dataset**: 10K+ element operations
- **test_variable_length_keys**: Keys of varying lengths
- **test_binary_keys_and_values**: Binary data handling

#### Iterator Tests
- **test_iterator_ordering**: Iterator order verification
- **test_range_iterator**: Range-based iteration

#### Edge Cases
- **test_empty_tree_operations**: Empty tree behavior
- **test_single_key_operations**: Single element operations
- **test_duplicate_insertions**: Duplicate key handling
- **test_common_prefix_handling**: Common prefix scenarios
- **test_node_transitions**: Node type transitions
- **test_memory_usage_tracking**: Memory tracking

#### Multi-Shard Tests
- **test_multiple_shards**: Shard isolation

### test_utils.rs

Utility functions for test data generation:
- `generate_sequential_keys()`: Sequential key generation
- `generate_random_keys()`: Random key generation
- `generate_sequential_kvs()`: Sequential key-value pairs
- `generate_random_kvs()`: Random key-value pairs
- `generate_prefix_keys()`: Keys with common prefixes
- `generate_node_growth_keys()`: Keys triggering node transitions
- `generate_variable_length_keys()`: Variable-length keys
- `create_temp_dir()`: Temporary directory creation
- `assert_bytes_eq()`: Byte slice comparison

## Running Tests

### Run All Tests
```bash
cargo test --package nanograph-art
```

### Run Specific Test File
```bash
# Basic unit tests
cargo test --package nanograph-art --test art_tests

# Integration tests
cargo test --package nanograph-art --test integration_tests
```

### Run Specific Test
```bash
cargo test --package nanograph-art test_kvstore_basic_put_get
```

### Run Tests with Output
```bash
cargo test --package nanograph-art -- --nocapture
```

### Run Tests in Parallel
```bash
cargo test --package nanograph-art -- --test-threads=4
```

### Run Tests with Specific Features
```bash
cargo test --package nanograph-art --all-features
```

## Test Data

Tests use various data patterns:
- **Sequential**: `key00000000`, `key00000001`, ...
- **Random**: Randomly generated byte sequences
- **Prefixed**: `common_prefix_000001`, ...
- **Binary**: Raw byte sequences including null bytes
- **Variable-length**: Keys from 1 to 20+ bytes

## Expected Behavior

### Insert Operations
- Should return `None` for new keys
- Should return old value for existing keys
- Should increment tree size
- Should trigger node growth when needed

### Get Operations
- Should return `Some(value)` for existing keys
- Should return `None` for missing keys
- Should not modify tree state

### Delete Operations
- Should return `Some(value)` for existing keys
- Should return `None` for missing keys
- Should decrement tree size
- Should trigger node shrinking when needed

### Iterator Operations
- Should visit all elements
- Should maintain consistent ordering
- Should handle empty trees
- Should support range queries

## Contributing

When adding new tests:
1. Use descriptive test names
2. Test both success and failure cases
3. Include edge cases
4. Use test utilities for data generation
5. Add comments for complex scenarios
6. Ensure tests are deterministic
7. Clean up resources (use temp directories)

## Test Organization

Tests are organized by:
- **Functionality**: Basic ops, range queries, iterators
- **Complexity**: Unit → Integration → Stress
- **Concurrency**: Single-threaded → Multi-threaded
- **Data Size**: Small (100) → Medium (1K) → Large (10K+)
